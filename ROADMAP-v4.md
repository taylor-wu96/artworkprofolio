# ROADMAP v4 — 從「一個有體溫的身體」到「一具誠實、好走、只說必要之話的身體」

> **產品迭代藍圖 v4**（2026-06-28）
> 配套：[ROADMAP.md](ROADMAP.md)（v1・瘦身/通電/擴音）／[ROADMAP-v2.md](ROADMAP-v2.md)（一體性＋護欄）／[ROADMAP-v3.md](ROADMAP-v3.md)（時間與材質的連續）／[DESIGN.md](DESIGN.md)（視覺設計）／[PROGRESS-REDESIGN.md](PROGRESS-REDESIGN.md)（工程落地）

> **這一輪的句子**：v3 之前的工作一直在「加」——加儀器、加呼吸、加光的記憶。v4 是第一次認真地「減」與「校準」。一座成熟的館，靠的不是它有多少件裝置，而是它**不擺多餘的東西**、**每扇門都好推**、**每句話都是真的**。

## 〇、為什麼有 v4 — 一個當代藝術家走進這座館，前五分鐘會說什麼

我把自己當成第一次來的策展同行，關掉所有文件，只看網站本身。誠實的話如下（不客氣，因為這是它應得的尊重）：

### 0.1 概念比作品大聲（最根本的批評）

這座館的**論述密度遠高於它的影像密度**。[DESIGN.md](DESIGN.md) 用四層當代藝術脈絡（Ikeda／Turrell／Steyerl／JODI）替每個像素背書；[ROADMAP-v3](ROADMAP-v3.md) 把「身體有沒有體溫」寫成詩。可是一個陌生人推門進來，他先遇到的是**一套很用力的觀看裝置**，而不是**一張讓他停下來的照片**。

當代藝術最忌諱的一件事，就是**鷹架比建築更好看**。你自己在 v3 §0.2 已經親口診斷了——「器官在空轉」「半亮的館比全暗更糟」。我同意，而且要說得更重一點：**這不是內容進度問題，這是優先序的世界觀問題。** 只要那批真照片還沒進來，所有的呼吸、殘光、準星、資料條，都是在替一個還沒到場的主角佈光。v4 的第一鐵律因此和 v3 一樣、但語氣更硬：**沒有作品之前，停止製造儀器。**

### 0.2 「資料精度」這層，正在說一個它還沒資格說的謊

[signature.js](src/lib/signature.js) 用 slug 當種子，生出座標、編碼、頻道，鋪在每件作品右下角（`data-strip`）。立意是 Ikeda／資料藝術。但 Ikeda 的資料是**真的**——那正是那件作品的全部倫理。**這裡的資料是生成的、是裝飾成資料的樣子。** 當你的整個概念脊椎宣稱「衰敗是時間的誠實」，卻在每件作品上貼一條**假的精密數據**，這是自我矛盾——你用一個關於「誠實」的隱喻，去支撐一個表演「精密」的謊。

一個當代藝術家會立刻嗅到這股「cosplay 資料藝術」的味道。**這比沒有資料條更傷。** 出路只有兩條，沒有第三條：(a) 讓資料變真（接真 EXIF/GPS，即 v3 的 C1 通血）；(b) 若這件作品沒有真資料，**就不要顯示資料條**。半真不是風格，是破綻。

### 0.3 它對「有滑鼠的桌機人」很迷人，對「拿手機的多數人」很薄

準星（reticle）、左右邊緣翻頁熱區、hover 轉綠、浮出頁邊的旁註——這套語彙的重心全在 **fine pointer**。但今天觀看藝術的主介面是**手機**。在手機上，準星不存在、hover 不存在、資料條被擠到換行（這正是你列的痛點 4）。也就是說：**這座館最精緻的部分，多數觀者根本看不到；他們看到的，是那些還沒被當成主場設計的退化版。** 一個在 2026 年辦展的藝術家，會把手機當成**主要的展牆**，而不是桌機的 fallback。

### 0.4 結論：v4 不加裝置，v4 校準這具身體

> v2 讓身體**連續地移動**；v3 給它**體溫、光的記憶、年歲**。
> v4 做三件 v1–v3 都沒做的事：**(1) 讓它只說真話**（資料誠實／schema 瘦身）、**(2) 讓每扇門都好推**（直覺修復：撞 navbar、放映版位、精選語意）、**(3) 讓生成影像這個器官真的能長大**（3D/sketch 可擴充性）。
> 工程仍是手段，審美仍是目的。**這一輪工程的角色是修剪刀與水平儀**——不是再裝一隻新的器官，是把已有的校正到位、把多餘的剪掉。

# H0 ｜直覺修復：先把每扇門推順（本週・零概念風險）

> 這三項都不動世界觀，只還債。它們是觀者「沒說出口但一直在扣分」的地方。先清，因為它們最便宜、最有感，而且擋在一切感官精修前面。

## B1 — 頂部 padding 撞 NavBar（痛點 2・根因已定位）

### 🔧 根因（不是表面，是孤兒樣式）
[components.css](src/styles/layers/components.css) 第 259 行有一條 `.work { padding-block: clamp(2.5rem,7vh,6rem) … }`——**但沒有任何頁面用 `.work` 這個 class。** 真正的內頁包在 `.detail-view.container > .artwork-layout / .gallery-layout / .sketch-view` 裡，而：
- `.artwork-layout`、`.gallery-layout`、`.sketch-view .artwork-layout`**完全沒有 `padding-block-start`**；`.work__header` 只有 `margin-bottom`。
- 只有 `.essay-layout` 自己帶了 `padding-block: clamp(3rem,8vh,6rem)`（所以散文頁沒事——這也解釋了為什麼是「某些頁面」）。
- [site-header](src/styles/layers/layout.css) 是 `position: sticky; top:0`，所以作品/影像/互動內頁的標頭（「作品 / Work」標籤＋大標）**直接貼在黏性導覽下緣**。

### 🧭 對策（一個 token，治根）
- 在 [tokens.css](src/styles/layers/tokens.css) 立 `--page-top: clamp(2.5rem, 7vh, 6rem)`（館的「進門退一步」）。
- `.detail-view`（目前 CSS 裡根本沒有這條規則，等於只吃 `padding-inline`）補 `padding-block-start: var(--page-top)`，讓**所有**內頁共用同一個進門距離；刪掉孤兒 `.work`。
- 順手治 in-page 錨點：首頁 `#works`（[index.astro](src/pages/index.astro)）被 sticky header 蓋住標題——給可被跳轉的標頭加 `scroll-margin-top: calc(header 高 + var(--space-md))`。
- **驗收**：作品／影像集／互動三種內頁的標頭與 navbar 之間有一致呼吸；跑 `npm run test:visual` 更新這三型 baseline；散文頁不動（已正確）。

## A1 — 放映模式：長圖說把影像頂上去（痛點 4・根因已定位）

### 🔧 根因（grid 的 `auto` 列在吃影像的 `1fr`）
[Lightbox](src/components/Lightbox.astro) 的 `.pjx` 是 `grid-template-rows: 1fr auto`（上＝影像舞台，下＝`.pjx__chrome`）。[media.css](src/styles/layers/media.css) 的 `.pjx__chrome` 是 `display:flex; flex-wrap:wrap`，裡面塞了 `count ＋ sig ＋ exif ＋ caption ＋ 說明鈕`。**圖說一長，chrome 換行變高，`auto` 列長高 → 壓縮 `1fr` 舞台 → 影像被迫縮小並上移。** 手機最嚴重（窄、字大、最容易換行），正是你的觀察。

### 🎨 藝術判斷：圖說不該是常駐的底欄
這違反你自己的鐵律「影像永遠是主角」。底欄該只放**短、定長、機器感**的東西（編號、簽名、EXIF——它們本就該是一行 mono）；**會變長的人話（caption / statement）屬於「說明 / Statement」呼出層**，不該擠壓影像。底欄是儀表，不是說明書。

### 🧭 對策（讓舞台永遠滿高）
- **把 chrome 從 grid 列改為覆蓋層**：`.pjx` 回到單層，`.pjx__stage` 滿版；`.pjx__chrome` 改 `position:absolute; bottom:0` ＋ 一道極淡 scrim（合法漸層：媒體層已有 scrim 前例 media.css:285）。影像高度從此**不被底欄左右**。
- **caption 移出常駐 chrome**：底欄只留 `count · sig · exif`（皆定長一行）；caption 併入「說明 / Statement」面板，或在底欄以**單行 ellipsis** 呈現、點擊才展開。
- 手機本就有 single-tap 切 `is-bare`（看純影像）——順勢讓底欄在手機預設更輕，長文一律走呼出。
- **驗收**：放映一張帶超長圖說的影像，桌機與手機影像位置/尺寸**不因圖說長度改變**；`reduced-motion` 不受影響；放映 VT 法律（E6）照跑。

## A3 — 「置頂精選」的語意決斷（痛點 5・這是策展決定，交給你）

### 🔧 根因（精選在和「時間即脊椎」打架）
[queries.js](src/lib/queries.js) 的 `LIST_ORDER = order(coalesce(featured,false) desc, publishedAt desc)`——精選先置頂，其餘依日期。但首頁（[index.astro](src/pages/index.astro)）的敘事是**「全部影像，依時間倒敘的檔案」**＋`year-marker` 年份分組＋編號 `total - i`＋追蹤真正最新的「最新 / Latest」標。於是矛盾發生：
- 一件**舊的**精選作被頂到第 1 位，但它落在它**該在的年份分組之外**——年份脊椎被打斷。
- 編號 `total - i` 與「最新」標各說各話（最新標貼在真正最新那件，未必是第 1 位）。
- 觀者讀到的是「依時間」，看到的卻不是純時間——**直覺被背叛。**

> 你說「不太確定置頂精選的意義在哪裡」——它現在確實沒有清楚的意義，而且在悄悄破壞檔案的核心隱喻（時間的誠實）。

### 🎨 三個選項（我推薦其一，最終是你的策展權）
- **選項 A（最純・我傾向這個）**：**移除 featured 排序**。讓時間成為唯一的軸。這最貼合「衰敗是時間的誠實」——館不替任何一件插隊。`featured` 欄位可留作他用或刪（見 B2）。
- **選項 B（保留但分流）**：檔案維持純倒敘；另闢一個**「此刻凝視 / Now」單格**置於 hero 與索引之間，只放**一件**當前主推作（不是一串精選）。精選＝策展的當下聲音，檔案＝時間的客觀記錄，兩者不互相污染。
- **選項 C（維持現狀）**：不建議——除非接受上述直覺背叛。

→ 我的建議：**A 為底，若你想要策展的當下聲音就加 B 的單格。** 這也順手回答了痛點 5 與痛點 6 的交集（featured 是否該留在 schema）。

# H1 ｜誠實化：讓這具身體只說真話（與 C1 通血同生・這一輪的靈魂）

> H0 把門推順了。H1 處理最深的批評（§0.2）：這座館正在說一些它還沒資格說的話。誠實化＝schema 只問每件作品真的有的東西、資料層只顯示真的數據。**這必須和 [ROADMAP-v3](ROADMAP-v3.md) 的 C1 通血綁在一起做**——因為「讓資料變真」與「沒真資料就別顯示」是同一枚硬幣。

## B2 — Schema 瘦身：「按作品類型給欄位」（痛點 6・你的核心直覺是對的）

### 🔧 現況盤點（從 [queries.js](src/lib/queries.js) 反推 schema）
單一 `post` 型別現在同時揹著：`title, slug, publishedAt, cover, category(artwork/gallery/essay), status(draft/wip/…), featured, capture{location,year,gear,film}, body, gallery[], themes[]`，外加 asset 內的真 EXIF。另有 `sketch`（engine/sketchId/aspectRatio/params/description）、`theme`、`series`、`about`。

問題正如你說的：**一個型別服務三種本質不同的東西（攝影作品／影像集／散文），於是每件作品都被問了一堆它不需要回答的問題。** 結果是欄位填得參差 → 顯示面參差 → **視覺一致性被 schema 的鬆散直接破壞**（這正是你目標 4「視覺一體性」的隱形殺手）。

### 🎨 原則：schema 的形狀，就是視覺的形狀
> 視覺一致，來自**填寫一致**；填寫一致，來自**schema 只問必要**。一個只問必要之事的後台，前台才可能長出一致的臉。

### 🧭 對策（兩步，先軟後硬）
- **第一步（低風險・先做）**：在現有 `post` 內用 Studio 的**條件欄位（hidden/readOnly by category）**收斂——攝影作品不顯示 `gallery[]` 的散文欄位、散文不問 `capture`/EXIF。前台 [queries.js](src/lib/queries.js) 對應地**只在該類型抓該欄位**。零資料遷移，先把「被問多餘問題」止血。
- **第二步（值得做・需遷移）**：把 `post` 拆成語意清楚的型別——`photograph`（單張／作品）、`gallery`（成捲影像集）、`essay`（散文）。各自只帶自己的欄位。`sketch` 已是獨立型別、是正確示範。前台的 `category` 分流邏輯（[post/[slug].astro](src/pages/post/[slug].astro) 那個三叉 if）會因此**從「執行期判斷」變成「型別本身」**，更乾淨、更難出錯。
- **連帶清理**：`featured`（看 A3 決定去留）、`status` 的取值是否真的都用到、`capture` 四欄是否每件都填得齊（填不齊就別顯示，見 B3）。
- **驗收**：每個型別的「必填／選填」清楚；前台不再抓不存在的欄位；Studio 重部署沿用 `schema-deploy-crash-workaround`（manifest extract ＋ `--no-extract-manifest` 繞 exit 134）。

## B3 — 資料層的誠實化（呼應 §0.2・半真簽名的去留）

### 🎨 藝術判斷
這是 v4 最該下決心的一刀。`data-strip`／`signature`／`glyph` 是漂亮的，但它們在**表演精密**。三條路：
- **誠實路（建議）**：資料條**只在有真資料時出現**——真 EXIF 來自照片、真 GPS 來自定位、真器材來自 `capture`。沒有，就讓那個角落乾淨。一件作品的右下角空著，比貼一串假座標更有尊嚴。
- **明示虛構路**：若你想保留生成式美學，就**坦白它是生成的**——例如把它定義為「館給每件作品的編目指紋（catalog fingerprint）」，是館的語言而非相機的數據。誠實的虛構是藝術，偽裝的虛構是破綻。
- **全撤路**：若上面兩條都不夠堅定，撤掉資料層，讓影像獨自承擔。克制本就是這套設計的核心。

### 🧭 對策
- 與 C1 通血同時：先讓**至少一件**作品走通真 EXIF（[signature.js](src/lib/signature.js) `readExif` 已就緒，缺的是有 EXIF 的真照片）。
- 改 `archiveSignature`：**真資料優先，無真資料則該欄不渲染**（而非用 slug 種子補滿）。`__sig` 綠 code 同理——它是「活訊號」，沒有真訊號就不該亮。
- **DoD 新增第九問**：*這個元素顯示的資訊，是真的嗎？如果不是真的，它有沒有坦白自己是生成的？*

# H2 ｜可擴充與行動主場（地基投資・內容量到門檻才全做）

## B4 — 生成作品的可擴充性（痛點 1・3D/sketch・根因已定位）

### 🔧 根因（契約是對的，周邊是空的）
[registry.ts](src/sketches/registry.ts) 目前只有兩件（`light-field` R3F、`drift-light` 原生 GL），手寫登錄。[types.ts](src/sketches/types.ts) 的 `SketchModule` 契約其實**設計得很好**（引擎無關、reducedMotion/mobile/dpr/params/scroll 都進了 ctx、有 pause/resume/destroy 生命週期）。可擴充性不足不在契約，在它**缺周邊**：
- **參數沒有型別與 UI**：`params` 是 Sanity 的 kv 字串陣列，前台 `JSON.parse` 容錯解析——作者要靠記憶填 key、沒有驗證、沒有預覽。
- **沒有 poster／fallback 自動化**：無 JS／載入前只有手填 `cover`；生成作品難有好的靜態封面（OG 立面、reduced-motion 靜態都吃虧）。
- **「React 只是眾引擎之一」還沒兌現**：Hero3D 是 R3F、drift-light 是 raw GL，但**沒有第三種引擎的範本**（p5/canvas2d/glsl）證明它真的引擎無關；新增一件＝從零寫。
- **沒有錯誤邊界**：一件 sketch 掛了，沒有優雅降級到 cover。

### 🎨 藝術判斷：把生成層接回館的語言
一個最漂亮的擴充，是讓**生成作品吃和資料層同一顆種子**——`signature.js` 的 slug 種子既然能生「指紋」，就能當生成作品的隨機源。於是「資料指紋」與「生成影像」是**同一個編目身分的兩種顯影**。這把 §0.2 的批評反轉成優勢：資料層不再是假裝的精密，而是**生成系統的真實輸入**。

### 🧭 對策（地基，分階段）
- **參數 schema 化**：每件 sketch 宣告自己的參數型別（name/type/default/range），Studio 端用它生表單、前台用它驗證——`params` 從「自由 kv」升為「有契約的旋鈕」。
- **poster 管線**：載入器在首次穩定幀 `captureFrame()` 存一張靜態圖，餵 OG／reduced-motion／fallback。一條指令離線預生。
- **第三引擎範本**：補一個 `canvas2d` 或 `p5` 範本作品，作為「新增一件＝照範本填空」的活文件——真正證明引擎無關。
- **錯誤邊界**：`mount` 失敗／WebGL context lost → 自動回落到 cover，不留白屏。
- **種子統一**：sketch 的 ctx 多一個 `seed`（來自 slug，與 `signature.js` 同源）。
- **驗收**：新增一件生成作品的步數從「寫模組＋改 registry＋手填 cover＋手填 params」降到「照範本填空＋一行 registry」；弱機/reduced-motion 有靜態 poster；一件壞掉不拖垮頁。
- **依賴**：內容門檻——生成作品 ≥3 件才值得做完整管線；否則先做錯誤邊界＋poster 兩項止血。

## A2 — 把手機當成主要的展牆（呼應 §0.3・重心轉移）
- **審計（先產出判斷，再動手）**：逐頁在真手機上走一遍，標出「桌機才有、手機退化」的每一處（準星、hover 綠、邊緣翻頁熱區、旁註浮邊、資料條換行）。
- **原則**：手機不是 fallback，是**主場**。凡是手機看不到的精緻（準星等），要嘛在手機有等價的觸覺替身，要嘛承認它是桌機限定的「房間彩蛋」而非核心語言——**別讓核心體驗依賴多數人沒有的輸入裝置。**
- 與 A1（放映版位）天然同生——放映是手機最重的觀看場景。

## A4 — 一致的留白與被感覺到的門檻（延續 v3 T1/T3 的收尾）
- v3 的 `--pulse` 母拍與 base.css 殘光（`body::before` radial-gradient）已落地——A4 是把**留白節奏**也收進 token：各頁標頭（`section-head` vs `detail-view`）的進門距離統一吃 `--page-top`（與 B1 同根），讓「每個房間的第一口呼吸」等長。
- 門檻線（v3 T3）若尚未落地，在此一併收尾——站名就是「門檻」，這是概念脊椎欠的最後一筆。

1. **B1 撞 navbar ＋ A1 放映版位**（H0）——最便宜、最有感、零概念風險，且 A1 直接修手機主場最痛的一處。**本週清掉。**

## 四、給每一次改動的 DoD（延續 v2 七問＋v3 第八問＋v4 第九問）

> **9. 這個元素顯示的東西是真的嗎？若不是真的，它有沒有坦白自己是生成的？**（呼應 §0.2 誠實化）
>
> 並重申 v3 第八問（任何新動態吃 `--pulse` 母拍、綠仍只在白名單）、v2 三條（兩房／手機／導覽過去）。
> 護欄延續：動全域過場/動畫必跑 `npm run test:visual` ＋ `npm run lint:design`；每條改動有 `reduced-motion` 靜態退回。
> **v4 加一條流程護欄**：凡屬「減法」（刪欄位／撤元素），先 spike 在一個頁面驗證「少了它是否更好」，確認後再全站施行——減法和加法一樣需要證據。
